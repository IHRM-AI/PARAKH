# PARAKH Health-Score Model Card

Version 0.1.0 · IDBI Innovate 2026, Track 3 (Financial Health Score) · Team IHRM

This card follows the structure banks expect for a model risk file: intended use,
model description, data, performance, calibration, limitations, fairness and
out-of-scope uses. It documents a prototype trained on a disclosed synthetic
population, not a production credit model.

## Intended use

- Primary use: originate New-to-Credit (NTC) MSMEs by turning consented alternate
  data (GST, Account Aggregator bank statements, EPFO) into a 0-100 financial
  health card with dimension sub-scores, vernacular reason codes and a
  pre-qualified working-capital limit.
- Role: decision-support only. Every card is a recommendation an underwriter
  reviews and approves; the model does not auto-sanction. The lender memo is
  explicitly gated on officer approval.
- Users: credit officers and underwriting teams at the sponsoring bank, operating
  on the bank's own book after in-sandbox retraining.

## Model

- Estimator: LightGBM gradient-boosted trees (binary objective) predicting a
  12-month default probability from 13 features across three consented sources.
- Calibration: isotonic regression fitted on a held-out validation fold maps the
  raw booster score to a probability of default.
- Score mapping: probability of default is mapped to a 0-100 health score with a
  points-to-double-the-odds (PDO) scale, anchored so that the reference odds
  (3:1 good-to-bad, PD = 0.25) sit at score 55 and every 15 points doubles the
  odds. The score is affine in the log-odds of repayment and clipped to a
  [10, 95] display band. See `src/parakh/scoring/model.py`.
- Explanations: SHAP TreeExplainer contributions are mapped to directional
  Hindi/English reason codes. See `src/parakh/interpret/reason_codes.py`.

## Data — synthetic disclosure

No public dataset joins GST, bank and EPFO records for the same firm, so the
demo corpus is generated by a disclosed synthetic persona engine
(`src/parakh/synth/persona.py`). The label is deliberately not a linear read-off
of the observed features: a hidden management latent moves default risk without
being directly measured, each source carries its own driver, and default risk
combines these through thresholds and interactions.

Because the labels are synthetic, absolute AUC on this corpus partly measures the
generator rather than real-world discrimination. The defensible signal is
therefore the **source-ablation ladder** and the model-family comparison, not the
headline AUC. All values below are read from `artifacts/ablation.json`.

## Performance

Source-ablation ladder (held-out test fold, out-of-sample AUC):

| Consented sources     | AUC   |
|-----------------------|-------|
| GST only              | 0.714 |
| + AA bank statements  | 0.736 |
| + EPFO                | 0.752 |

Each consented source adds genuine, incremental discrimination. This ladder is
the primary result: it holds regardless of the generator's absolute difficulty
because the same generator produces every rung.

Model-family comparison (full feature set, same fold):

| Model               | AUC   |
|---------------------|-------|
| Logistic regression | 0.735 |
| LightGBM (GBM)      | 0.752 |

The gradient-boosted model beats logistic regression by +0.017 AUC, consistent
with the thresholds and interactions built into the generator.

Split-level metrics:

- Random split (n = 1600, default rate 0.245): AUC 0.752, Gini 0.505, KS 0.387,
  Brier 0.156.
- Out-of-time split (n = 2636, default rate 0.277): AUC 0.718 versus 0.752 on the
  random split. Later origination cohorts face a harsher macro backdrop, so the
  out-of-time drop is expected and is the honest number to plan against.

## Calibration

Isotonic regression is fitted on the validation fold, so predicted probabilities
are monotone in the raw score and calibrated to the observed default rate on that
fold. Brier scores are reported above (0.156 random, 0.176 out-of-time). The PDO
mapping is a deterministic, documented transform of the calibrated probability
and introduces no additional fitting.

## Limitations

- Synthetic corpus: trained and evaluated on a generated population. Absolute AUC
  partly reflects the generator; only the ablation ladder and family comparison
  are claimed as evidence of real signal.
- No live rails: there is no live Account Aggregator, GST or EPFO integration in
  this prototype. Adapters are schema-exact so production rails swap in as
  configuration, but the data here is synthetic.
- Monitoring is a re-score, not a forecaster: the 12-month monitoring trajectory
  re-scores simulated monthly snapshots through the same model
  (`src/parakh/monitoring/trajectory.py`). It is a deterministic reconstruction
  for the demo, not a trained temporal early-warning model.
- Retraining required: any production use must retrain on the bank's own book in
  the sandbox before the numbers here transfer.

## Fairness stance

The model consumes financial-behaviour features only (turnover, filing
punctuality, cash buffer, bounces, headcount trend, GST-vs-bank gap). It ingests
no protected attributes such as caste, religion, gender or region as a
score driver (state and sector exist in the generator for realism but are not
model inputs). Reason codes are surfaced in Hindi and English so borrowers can
understand and contest their score. Formal disparate-impact testing on protected
groups is out of scope for the synthetic prototype and is a prerequisite before
any production deployment.

## Out-of-scope uses

- Automated, human-out-of-the-loop credit decisions or sanctions.
- Consumer or retail lending; the model is scoped to MSME working-capital
  origination.
- Any use of the synthetic-fold metrics as a production performance claim.
- Deployment without in-sandbox retraining on real, consented data and without
  the underwriter approval step.
